# Laravel 13 Modular API Framework

A high-performance, **Headless API-Only Backend** designed for enterprise-grade applications. This architecture focuses on scalability, modularity, transaction safety, and strict auditability.

## 🚀 Key Features

*   **Modular Monolith**: Organized by functional modules (Users, Products, Invoices) to keep the codebase maintainable and decoupled.
*   **Job-Centric Logic**: All business actions are encapsulated in Transactional Jobs.
*   **API Versioning**: Native support for URL-based versioning (e.g., `/api/v1/`, `/api/v2/`).
*   **Database Idempotency**: Built-in protection against duplicate requests using `Idempotency-Key` headers.
*   **Automatic Audit Logging**: Tracks every field change and job execution status.
*   **Response Macros**: Unified JSON response structure for success, errors, and validation.

## 📂 Project Structure

```text
app/
├── Abstracts/             # Base Job, ModuleServiceProvider, and SearchJob
├── Http/
│   ├── Controllers/       # Base Controller with transformIndex
│   └── Middleware/        # IdempotencyMiddleware
├── Interfaces/            # Job Contexts (ShouldCreate, ShouldUpdate, ShouldDelete)
├── Modules/               # Functional Modules
│   └── [ModuleName]/
│       ├── Http/          # Versioned Controllers (V1, V2)
│       ├── Jobs/          # Business Logic
│       ├── Routes/        # Versioned Routes (v1.php, v2.php)
│       └── Providers/     # Module Service Provider
└── Traits/                # Reusable logic (Relationships, Jobs, Audit)
```

### 📂 Module Structure
Organize your controllers by version within the module:
```text
app/Modules/User/
├── Http/
│   └── Controllers/
│       ├── V1/
│       │   └── Users.php
│       └── V2/
│           └── Users.php
├── Routes/
│   ├── v1.php
│   └── v2.php
```

### 2. Implementing a Job
- Extend `App\Abstracts\Job` to ensure the logic is validated, authorized, and audited automatically.
```php
class CreateProduct extends Job implements ShouldCreate 
{
    public function rules(): array {
        return ['name' => 'required', 'price' => 'required|numeric'];
    }

    protected function execute() {
        return Product::create($this->request->all());
    }
}
```
- Dispatching from Controllers
Keep controllers clean by using the `transformIndex` for lists and `dispatch` for actions.
```php
public function index() {
     // Just pass the query. The Job handles search, sort, and paginate.
    return $this->transformIndex(Product::query());
}

public function store(Request $request) {
    $data = $this->dispatch(new CreateProduct($request));
    return response()->success($data, 'Product Created', 201);
}
```


2. Update the ModuleServiceProvider
Modify your base provider to automatically prefix routes with the version found in the filename.
```php
// app/Abstracts/ModuleServiceProvider.php
protected function registerRoutes(): void
{
    $versions = ['v1', 'v2']; // Or scan the Routes directory

    foreach ($versions as $version) {
        $routeFile = "{$this->basePath}/Routes/{$version}.php";

        if (file_exists($routeFile)) {
            Route::prefix("api/{$version}/{$this->alias}")
                ->middleware(['api', 'idempotent']) // Global API logic
                ->name("api.{$version}.{$this->alias}.")
                ->group($routeFile);
        }
    }
}

3. Route Definition (Routes/v1.php)
The routes become extremely clean because the prefixing is handled by the provider.
php
// app/Modules/User/Routes/v1.php
use App\Modules\User\Http\Controllers\V1\Users;
use Illuminate\Support\Facades\Route;

Route::get('/', [Users::class, 'index']);
Route::post('/', [Users::class, 'store']);
Use code with caution.
4. Controller Versioning
You can inherit logic from V1 into V2 and only override what changes.
php
// app/Modules/User/Http/Controllers/V2/Users.php
namespace App\Modules\User\Http\Controllers\V2;

use App\Modules\User\Http\Controllers\V1\Users as V1Users;

class Users extends V1Users
{
    public function index()
    {
        // V2 might return extra fields or use a different SearchJob
        return parent::index();
    }
}

5. URL Access

POST /api/v1/user (with Idempotency-Key)
```

### 3. API Headers (Idempotency)
All `POST`, `PUT`, and `DELETE` requests must include a unique UUID:\
`Idempotency-Key: 7b2d5f1a-3c4e-4a1b-9d2f-8e5c6a1b2c3d` `<UUID-V4>`

### 4. Response Macros
Your controllers should use the built-in macros for consistency:
```php
return response()->success($data, 'Action Successful', 201);
return response()->error('Unauthorized', 403);
return response()->validationError($errors);
```
📡 API Response Formats

### Response Format
All responses follow this macro-driven structure:\
- Success (200/201)

```json
{
    "success": true,
    "message": "Action completed successfully",
    "data": { ... }
}
```
-Error (403)
```json
{
    "success": false,
    "message": "Unauthorized",
    "errors": {
        "email": ["The email has already been taken."]
    }
}
```
-Validation Error (422)
```json
{
    "success": false,
    "message": "Validation failed",
    "errors": {
        "email": ["The email has already been taken."]
    }
}
```

2. Protecting the Versions
You can apply different middleware to different versions if needed (e.g., v2 might require a more modern auth scheme).
php
// app/Abstracts/ModuleServiceProvider.php
protected function registerRoutes(): void
{
    $versions = ['v1', 'v2'];

    foreach ($versions as $version) {
        $routeFile = "{$this->basePath}/Routes/{$version}.php";

        if (file_exists($routeFile)) {
            Route::prefix("api/{$version}/{$this->alias}")
                ->middleware(['api', 'auth:sanctum', 'idempotent']) // Forced security for all versions
                ->group($routeFile);
        }
    }
}
Use code with caution.
3. Handling "Deprecation"
When you move to V2, you don't need to rewrite V1. You can simply add a header to V1 responses to warn frontend teams:
php
// In your V1 Controller
public function index()
{
    return parent::index()->header('X-API-Deprecated', 'v1 is deprecated. Please move to v2 by Dec 2024.');
}
